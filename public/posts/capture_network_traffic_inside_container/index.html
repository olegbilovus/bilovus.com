Capture Network Traffic Inside a Container

Network traffic analysis within containerized environments presents unique challenges due to the isolation mechanisms employed by container runtimes. This guide examines an approach to capturing and analyzing network traffic originating from Docker containers using tcpdump and Wireshark.

The methodology described employs a host-based packet capture strategy, wherein tcpdump operates on the host machine to monitor traffic traversing the virtual network interfaces associated with container instances. Real-time analysis is facilitated through Wireshark’s SSH remote capture functionality, which establishes a secure tunnel between the analyst’s workstation and the host system.

For demonstration purposes, this guide utilizes go-httpbin, an HTTP testing service that implements various endpoints conforming to the httpbin specification. This container serves as a representative example of network traffic generation and inspection.

System Requirements

The following prerequisites must be satisfied to perform packet capture on containerized traffic:

  • SSH access to the host machine running the container runtime
  • Root or sudo privileges on the host system
  • tcpdump utility installed on the host machine
  • Network interface identification tools (ifconfig or ip)
  • Text filtering utilities (grep) for interface discovery

Demonstration Environment Setup

To isolate the target container’s traffic from other network activity, a dedicated bridge network should be created. This approach simplifies the identification of the relevant virtual network interface and reduces noise in the captured data.

Execute the following commands to establish the demonstration environment:

1
2
3

docker network create go-httpbin-net

docker run --name go-httpbin -p 8090:8080 --network go-httpbin-net -d ghcr.io/mccutchen/go-httpbin

This configuration creates an isolated network namespace for the go-httpbin container, binding the container’s port 8080 to the host’s port 8090.

Identifying the Container’s Virtual Network Interface

Docker containers communicate through virtual network interfaces created by the container runtime. To capture traffic specific to a container, the corresponding host-side virtual interface must be identified.

The identification process begins by determining the container’s network gateway address:

1

docker inspect -f '{{range .NetworkSettings.Networks}}{{.Gateway}}{{end}}' go-httpbin

This command queries the container’s network configuration and returns the gateway IP address assigned to the virtual bridge network. In this example, the expected output is 172.19.0.1, corresponding to the go-httpbin-net network.

Once the gateway address is known, the associated network interface can be located using standard network inspection tools:

1

ifconfig | grep -B1 "172.19.0.1"

The output typically reveals a bridge interface with a naming pattern of br-<network_id>:

1
2

br-299a8e72a394: flags=4163<UP,BROADCAST,RUNNING,MULTICAST>  mtu 1500
        inet 172.19.0.1  netmask 255.255.0.0  broadcast 172.19.255.255

In this instance, br-299a8e72a394 represents the virtual bridge interface through which the container’s network traffic is routed. Note that containers using Docker’s default bridge network will typically show the interface name docker0.

Configuring Wireshark for Remote Packet Capture

Wireshark supports remote packet capture through its sshdump extcap interface, which enables packet capture on remote systems via SSH. This functionality must be explicitly enabled during Wireshark installation on Windows systems.

Capture Configuration Procedure

The following steps establish a remote capture session:

  1. Launch Wireshark and open the Capture Options dialog (Ctrl + K).
  2. Locate the “SSH remote capture: sshdump” interface in the capture interfaces list.
  3. Select the configuration icon (gear icon) adjacent to the SSH interface entry.
  4. In the “Server” and “Authentication” sections, provide the SSH connection credentials and hostname for the container host.
  5. Configure the capture parameters in the “Capture” tab:
    1. Set the “Remote interface” field to the identified bridge interface (e.g., br-299a8e72a394).
    2. Select “tcpdump” as the “Remote capture command selection”.
    3. Enable the “Use sudo on the remote machine” option, as packet capture operations require elevated privileges.
    4. Apply the configuration and initiate the capture session.

Upon successful connection, Wireshark will display real-time packet data from the specified container network interface. Traffic generated by accessing the go-httpbin service will appear in the packet list.

Performance Considerations

Packet capture operations introduce overhead that may affect system performance. Key considerations include:

  • CPU Utilization: The tcpdump process consumes CPU cycles for packet copying and filtering. High-throughput environments may experience measurable performance degradation.
  • Disk I/O: Writing captures to disk generates I/O load. For extended captures, ensure adequate disk space and consider using capture filters to reduce data volume.
  • Network Latency: While minimal, the packet capture mechanism may introduce microsecond-scale latency variations.

To minimize performance impact, employ BPF (Berkeley Packet Filter) expressions to filter packets at the kernel level:

1

tcpdump -i br-299a8e72a394 -n 'tcp port 8080' -w capture.pcap
